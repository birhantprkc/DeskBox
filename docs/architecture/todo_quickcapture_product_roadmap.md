# DeskBox 随记 / 待办产品路线图

本文档用于收敛 DeskBox `随记（Quick Capture）` 与 `待办（Todo）` 的后续产品方向，重点回答四件事：

1. 现在这两个功能已经做到什么程度。
2. 和成熟竞品相比，真正差在哪。
3. 用户高频痛点里，哪些值得补，哪些不值得学重。
4. 后续应该按什么顺序做，才能保持 DeskBox 轻量、原生、桌面化的路线。

如果后续功能边界、数据结构、交互模式或版本节奏有调整，需要同步更新本文档，避免产品判断、开发实现和对外表述逐渐跑偏。

## 一、产品定位

### 1. 待办

DeskBox 的待办不应该走“全能任务管理器”路线。

它的核心价值不是项目管理，而是：

- 常驻桌面，低摩擦记录
- 比便签更强，比大型 Todo 更轻
- 能快速看见今天要做什么
- 能和桌面文件、随记、音乐等上下文放在一起使用

一句话定义：

> `桌面轻待办`，而不是另一个 TickTick / Todoist。

### 2. 随记

DeskBox 的随记不应该走“知识库系统”路线。

它的核心价值不是长期知识管理，而是：

- 随手记
- 随手存
- 随手拖
- 稍后再整理

它更像桌面上的 `捕获收件箱 + 轻量缓存层`。

一句话定义：

> `桌面捕获中心`，而不是另一个 Obsidian / OneNote。

## 二、当前能力盘点

以下盘点基于当前仓库代码整理，目的是确认“已经有的能力”，避免后续把现成能力误判成待补缺口。

### 1. 待办当前已具备

- 快速添加
- 全屏编辑 / 内联编辑
- 编辑、删除、撤销
- 完成 / 取消完成
- 重要标记
- 颜色标记
- 筛选：全部 / 进行中 / 今天 / 重要 / 已完成
- 截止日期时间
- 快捷截止项：今天 / 明天 / 本周 / 清除
- 逾期状态与逾期文案
- 已完成项自动下沉，未完成优先
- 已完成视觉弱化与删除线
- 原生提醒服务
- 全局提醒开关与默认提醒提前量
- 单击复制文案
- 多选复制
- Ctrl / Shift 多选
- Ctrl + A / Ctrl + C / Esc
- 鼠标空白区框选
- 多条文本标准化复制
- 选中项拖拽导出文本
- 外部文本拖入生成待办
- 随记文本拖入生成待办

### 2. 待办当前仍然明显缺失

- 重复任务
- 每条任务独立提醒策略
- 稍后提醒（Snooze）
- 子步骤 / 子任务
- 日 / 周规划视图
- 分组 / 小节 / 清单维度
- 智能筛选保存
- 批量改时间 / 批量改颜色 / 批量完成

### 3. 随记当前已具备

- 记录 / 置顶 / 最近 三类视图
- 搜索
- 内联输入
- 全屏编辑
- 自动剪贴板捕获文本
- 自动剪贴板捕获图片
- 最近记录上限控制
- 最近记录保存到正式记录
- 置顶 / 取消置顶
- 置顶排序
- 文本 / 链接 / 图片 三类内容
- 图片缓存
- 缩略图生成
- OCR 提取图片文字
- 单条复制
- 多选复制
- 多选拖拽文本导出
- Ctrl / Shift 多选
- Ctrl + A / Ctrl + C / Esc
- 鼠标空白区框选
- 外部拖入文本 / 链接 / 图片 / 文件内容
- 待办文本拖入生成随记

### 4. 随记当前仍然明显缺失

- 标签 / Label
- 更强的筛选器
- OCR 内容纳入搜索索引
- 显式“转为待办”
- 显式“稍后提醒”
- 每日收件箱 / 时间线回看
- 轻模板
- 更强的整理入口

## 三、和成熟竞品对比

这里不是为了追求“功能对齐”，而是为了识别哪些能力已经被市场证明是高频刚需。

### 1. 待办方向对比

#### Microsoft To Do

强项通常在：

- 重复任务
- 提醒
- 步骤 / 子步骤
- 清单组织
- 微软生态联动

DeskBox 相比它的优势：

- 桌面驻留更自然
- 输入与查看更快
- 更贴近桌面文件工作流

DeskBox 相比它的短板：

- 规划性不够
- 结构层次不够
- 任务生命周期能力还偏浅

#### Todoist

强项通常在：

- 快速录入
- 重复规则
- 标签 / 项目 / 过滤器
- 提醒体系
- 视图成熟

DeskBox 相比它的优势：

- 更轻
- 更原生
- 更适合 Windows 桌面常驻场景

DeskBox 相比它的短板：

- 缺重复任务
- 缺项目 / 标签组织
- 缺更成熟的时间规划视图

#### TickTick

强项通常在：

- 日历
- 习惯
- 番茄钟
- 任务与时间结合

DeskBox 不建议照着补齐它的重功能，但它证明了一件事：

> 用户一旦开始认真使用 Todo，很快就会从“记下来”进入“怎么安排今天”。

所以 DeskBox 的待办，迟早需要一个轻量规划视图，而不只是列表。

### 2. 随记方向对比

#### Google Keep

强项通常在：

- 极低摩擦记录
- 标签
- 提醒
- 协作
- 搜索与轻组织

DeskBox 相比它的优势：

- 桌面常驻
- 和本地文件、拖拽、桌面上下文结合更紧
- 图片拖拽、OCR、本地缓存更适合 Windows 场景

DeskBox 相比它的短板：

- 缺标签
- 缺提醒
- 缺长期回找的组织能力

#### Sticky Notes

强项通常在：

- 原生感
- 足够轻
- 低学习成本

DeskBox 当前其实已经在“轻便但不简陋”这条线上比它走得更远，但也要注意不要越做越重，最后失去这种轻松感。

#### OneNote / Obsidian

这两类产品证明了另外一件事：

> 记笔记最大的难点，不是写下，而是几年后还能找回来、继续用。

DeskBox 不需要做成它们那样的结构化系统，但必须补一些最轻量的“可回找能力”。

## 四、从社区痛点看，用户真正缺什么

以下结论参考了官方产品页面与公开社区讨论，时间点为 `2026-07-08`。这些讨论不代表严格统计结果，但足够反映高频需求方向。

### 1. 任务类产品高频痛点

- 任务能记，但不好安排到今天
- 缺重复任务
- 缺每任务提醒或提醒粒度太粗
- 缺子任务 / 步骤
- 批量操作不顺
- 过滤器不够智能
- “轻任务”工具太轻，“重任务”工具太重，中间层产品很少

### 2. 随记类产品高频痛点

- 记得快，但以后找不到
- 图片越来越多后，卡顿、搜索差、组织弱
- 只有置顶和搜索，不够长期整理
- 很多记录其实是“稍后处理”，但没有提醒或转任务通道
- 太重的笔记软件不适合碎片捕获，太轻的便签又缺组织能力

## 五、DeskBox 最值得补的能力

这里按价值、风险、与你当前路线的契合度来排优先级。

### P1：高价值、强契合、应该优先做

#### 1. 待办：重复任务

这是待办从“能用”走向“长期留存”的关键能力。

建议第一版只做：

- 每天
- 每周
- 每月
- 工作日
- 自定义间隔（简化版）

先不做复杂 RRULE。

#### 2. 待办：每条任务独立提醒

当前更像“全局提醒偏移”，后续建议支持：

- 不提醒
- 准时
- 提前 5 分钟
- 提前 10 分钟
- 提前 30 分钟
- 提前 1 小时
- 提前 1 天

这会比单一全局设置实用得多。

#### 3. 待办：稍后提醒

原生通知或右键里支持：

- 10 分钟后提醒
- 30 分钟后提醒
- 明天提醒

这个能力比做复杂项目管理更贴近桌面使用场景。

#### 4. 随记：标签系统

这是随记最值得补的第一能力。

建议不要做文件夹树，第一版做轻量标签即可：

- 每条随记 0~N 个标签
- 标签筛选
- 常用标签建议
- 最近使用标签优先

#### 5. 随记：OCR 内容进入搜索

现在 OCR 只能“提取”，但未来更应该“可检索”。

这会显著提升图片型随记的长期价值。

#### 6. 随记：显式转待办 / 稍后处理

你现在已经有拖拽转化，这是好基础。

下一步应该加显式动作：

- 转为待办
- 转为重要待办
- 稍后提醒

这样用户不需要先知道拖拽逻辑，也能发现这条能力。

### P2：有价值，但应在 P1 之后

#### 1. 待办：轻量 Today 规划视图

不是完整日历，而是：

- 今天到期
- 接下来几小时
- 晚些再做

更像一个轻 Agenda。

#### 2. 待办：子步骤

适合复杂但不够复杂到要拆成多个任务的场景。

#### 3. 待办：智能筛选保存

例如：

- 今天 + 未完成
- 重要 + 逾期
- 无截止时间

#### 4. 随记：每日收件箱 / 时间线

适合回看今天捕获了什么，但不建议一上来做成大而全日志系统。

#### 5. 随记：轻模板

例如：

- 电话记录
- 会议摘记
- 灵感
- 稍后处理

### P3：可以观察，不要急着做

#### 1. 更重的项目管理

例如看板、负责人、复杂项目、跨列表依赖等。

不适合当前阶段。

#### 2. 笔记双链 / 知识图谱

这会把随记从捕获工具推向知识库产品，边界会立刻变重。

#### 3. 云同步 / 协作

价值大，但产品、账户、冲突处理、渠道策略和隐私责任都会陡增。

不适合现在插进主线。

## 六、明确不建议现在做的东西

为了避免产品越长越重，下面这些我建议主动克制：

- 不把待办做成完整项目管理器
- 不把随记做成 Markdown 知识库
- 不一上来引入复杂同步
- 不为了“和竞品一样”而硬塞重型视图
- 不用一堆第三方生产力概念堆功能密度

DeskBox 的竞争力不在“大而全”，而在：

- 桌面常驻
- 原生感
- 轻量
- 快速进入和离开
- 与 Windows 桌面工作流贴合

## 七、推荐开发顺序

### 阶段 A：先把两个核心短板补齐

1. 待办重复任务
2. 待办每任务独立提醒
3. 待办稍后提醒
4. 随记标签
5. OCR 文本纳入搜索
6. 随记显式转待办

### 阶段 B：提升长期使用效率

1. 待办轻 Agenda
2. 待办子步骤
3. 随记每日收件箱
4. 随记轻模板
5. 双方更强批量操作

### 阶段 C：视反馈决定要不要走更深

1. 智能筛选
2. 更强组织模型
3. 跨设备 / 账号能力

## 八、实现建议与风险提醒

### 1. 待办重复任务

建议先在数据层增加简单字段，而不是直接做复杂规则表达式。

例如：

```csharp
public string RepeatMode { get; set; } = "None";
public int RepeatInterval { get; set; } = 1;
public DayOfWeek[]? RepeatDays { get; set; }
```

第一版只要能稳定生成下一次到期时间，就足够了。

### 2. 每任务提醒

建议任务本身记录提醒配置，不要继续全部依赖全局默认值。

全局设置保留为默认创建值即可。

### 3. OCR 搜索

不要在展示时临时 OCR 搜索。

正确方向是：

1. 图片导入或 OCR 时缓存文本
2. 把 OCR 文本写入随记模型
3. 搜索时走统一文本索引

这样性能和体验才稳定。

### 4. 标签系统

建议先做“自由标签 + 最近常用”，不要一开始做复杂标签管理页面。

### 5. 转待办 / 稍后处理

建议抽象成统一动作层，避免未来出现：

- 拖拽能转
- 菜单能转
- 快捷键不能转

三套行为各写一套。

## 九、最终建议

如果只从“用户最需要、对 DeskBox 最有价值、又不会把产品做重”这三个维度交叉来看，我建议后续主线优先做：

1. `待办重复任务`
2. `待办每任务提醒 + 稍后提醒`
3. `随记标签`
4. `OCR 搜索化`
5. `随记显式转待办`

这五项完成后，DeskBox 的随记和待办会从“已经很好用的小组件”，提升到“足够让人长期放在桌面上每天用”的阶段。
