# DeskBox 待办重复任务设计

本文档用于定义 DeskBox `待办重复任务` 的第一阶段实现方案，目标是：

1. 保持当前 Todo 的轻量、原生、桌面化体验。
2. 在不引入重型项目管理复杂度的前提下，补齐高频刚需。
3. 让后续的“每任务提醒 / 稍后提醒 / 子步骤”可以自然接上。

如果后续重复规则、提醒逻辑、数据结构或交互入口有调整，需要同步更新本文档，避免开发实现和产品预期逐渐偏离。

## 一、目标与原则

### 1. 产品目标

DeskBox 的重复任务，不是为了追求和 Todoist / TickTick 同级别规则系统，而是为了解决最常见的桌面场景：

- 每天要做的事
- 每周固定要做的事
- 每月固定要做的事
- 工作日循环的事

第一阶段的成功标准不是“规则很多”，而是：

- 用户能快速设置
- 完成后能稳定滚到下一次
- 不会生成一堆立刻逾期的脏数据
- 不破坏当前排序、提醒、完成态和复制逻辑

### 2. 设计原则

- 优先做高频规则，不做复杂 RRULE
- 优先保持当前 Todo 列表模型，不额外引入大型视图
- 优先使用现有右键 / 轻弹层交互，不改成重面板
- 优先保证数据可迁移、可扩展，而不是第一版字段最少

## 二、范围定义

## 1. 第一阶段要做什么

第一阶段支持以下重复模式：

- `None`：不重复
- `Daily`：每天
- `Weekly`：每周
- `Monthly`：每月
- `Weekdays`：工作日

第一阶段行为要求：

- 重复任务必须依附于截止时间
- 标记完成后，自动生成下一次待办
- 当前这次保留为已完成记录
- 下一次保留原有文案、颜色、重要标记
- 下一次重置完成状态与提醒状态

## 2. 第一阶段明确不做

- 自定义 RRULE
- 每 N 周并指定多个星期几
- 每月第 N 个星期几
- 结束日期 / 重复次数上限
- 跳过一次
- 编辑单次 / 编辑整组的双分支弹窗
- 重复任务专属日历视图

这些能力后续可以扩，但不应该进入第一阶段。

## 三、交互方案

### 1. 设置入口

建议第一阶段只放在任务右键菜单内，位置紧跟在 `截止日期` 下方：

- 截止日期
- 重复
  - 不重复
  - 每天
  - 每周
  - 每月
  - 工作日

原因：

- 风险最低
- 不需要重做当前输入区
- 和现有 `截止日期` 心智一致
- 最符合你现在这套原生右键菜单路径

### 2. 展示方式

建议不要新增大块 UI。

第一阶段只做轻提示：

- 在日期状态文案前增加重复说明
- 样式示例：
  - `每天 · 今天 18:00`
  - `每周 · 明天 09:00`
  - `工作日 · 已逾期`

如果文案长度过长，可退化成：

- `重复 · 今天 18:00`

但更建议保留具体模式，用户一眼能判断规则。

### 3. 编辑行为

第一阶段不做“这一次 / 之后全部”的复杂选择。

规则如下：

- 改截止时间：只影响当前任务及它未来生成出的后续任务
- 改重复模式：只影响当前任务及它未来生成出的后续任务
- 清除截止时间：自动清除重复模式

这个语义简单，也符合 DeskBox 当前轻量定位。

## 四、数据结构设计

### 1. 推荐模型

不建议把重复字段全部平铺到 `TodoItem` 顶层。

更适合新增一个轻量子对象，避免未来字段继续膨胀：

```csharp
public sealed class TodoRecurrence
{
    public string Mode { get; set; } = TodoRecurrenceMode.None;
    public DateTimeOffset? AnchorDueDate { get; set; }
}

public static class TodoRecurrenceMode
{
    public const string None = "none";
    public const string Daily = "daily";
    public const string Weekly = "weekly";
    public const string Monthly = "monthly";
    public const string Weekdays = "weekdays";
}
```

`TodoItem` 新增：

```csharp
public TodoRecurrence? Recurrence { get; set; }
```

### 2. 为什么保留 `AnchorDueDate`

`AnchorDueDate` 的作用不是当前展示，而是为了以后规则扩展和稳定计算。

它至少有两个价值：

1. 后续如果支持“编辑当前任务后延续到未来”，有一个稳定锚点可用。
2. 月度重复在跨月时更容易保持预期规则。

第一阶段虽然可以仅靠 `DueDate` 计算，但保留锚点更稳。

### 3. 数据版本

建议把 `TodoWidgetData.Version` 从 `1` 升到 `2`。

迁移规则：

- 老数据没有 `Recurrence` 时，按不重复处理
- 已有 `DueDate`、`CompletedAt`、提醒字段保持原样
- 新字段缺失时自动补默认值

### 4. Normalize 约束

建议在 `TodoWidgetStore.Normalize()` 内新增：

- `Recurrence` 为 `null` 时视为不重复
- `Mode` 非法时归一为 `none`
- 若 `DueDate == null`，则 `Recurrence = null`
- 若 `Mode == none`，则 `Recurrence = null`
- `AnchorDueDate == null && DueDate != null` 时，可回填当前 `DueDate`

## 五、完成后的推进规则

这是整个功能最关键的部分。

### 1. 基础规则

当一个重复任务被标记为完成时：

1. 保留当前任务为已完成
2. 基于当前任务生成一条新的未完成任务
3. 新任务的 `DueDate` 推进到下一次
4. 新任务继承原文案、颜色、重要标记、重复规则
5. 新任务重置完成与提醒状态

### 2. 为什么不直接把原任务“滚动到下一次”

因为那样会丢失用户刚刚完成这次任务的反馈。

而你当前 Todo 已经有：

- 已完成列表
- 完成时间
- 完成态视觉

保留这次记录，体验更一致，也更符合现有模型。

### 3. 下一次时间的计算规则

必须避免一个问题：

> 用户今天才完成一条上周就逾期的重复任务，结果系统只推进一步，新任务立刻继续逾期。

因此建议规则为：

- 从当前 `DueDate` 开始推进
- 一直推进到 `nextDue > completionTime`
- 再把这个 `nextDue` 作为下一条任务的截止时间

这意味着：

- 日重复：补到完成时刻之后的下一天
- 周重复：补到完成时刻之后的下一周对应日
- 月重复：补到完成时刻之后的下一月对应日
- 工作日：推进到下一个工作日

### 4. 月重复规则

月重复需要定义边界：

- 如果原时间是 `1 月 31 日 18:00`
- 下一次遇到没有 31 日的月份

建议规则：

- 自动夹取到该月最后一天
- 保留时分秒

示例：

- `1/31 18:00` -> `2/28 18:00`
- `3/31 18:00` -> `4/30 18:00`

### 5. 工作日规则

`Weekdays` 定义为：

- 周一至周五

推进时跳过：

- 周六
- 周日

第一阶段不处理法定节假日，这部分后续若做就属于更高复杂度能力。

## 六、提醒联动规则

当前提醒服务是按：

- 全局开关
- 全局提前量
- 每条任务的 `DueDate`

来触发的。

重复任务第一阶段建议这样接：

### 1. 当前完成项

当重复任务被完成：

- 当前已完成任务不再提醒

这符合现有逻辑。

### 2. 新生成项

新生成任务需要：

- `ReminderLastNotifiedAt = null`
- `ReminderDismissedForDueDate = null`

也就是说，它是一次全新的待提醒任务。

### 3. 提醒时间计算

第一阶段继续沿用全局提醒提前量即可，不需要和重复任务额外耦合。

后续如果做“每任务独立提醒”，则新生成任务应继承原任务的提醒配置。

## 七、排序与列表行为

### 1. 当前任务完成后

当前任务进入已完成区，沿用现有逻辑：

- 已完成在下
- 按完成时间排序

### 2. 新生成任务

新生成任务进入未完成区，沿用现有排序：

- 未完成在上
- 按截止时间从近到远
- 无截止时间放后

### 3. SortOrder 建议

由于当前未完成项主要受截止时间排序影响，重复生成项不需要强行插到原位置。

建议：

- 新生成项默认插入 `Items`
- 通过现有刷新逻辑参与排序
- `SortOrder` 只作为兜底，不作为重复项定位主依据

## 八、UI 与状态文案建议

### 1. 本地化键建议新增

- `Todo.Recurrence.None`
- `Todo.Recurrence.Daily`
- `Todo.Recurrence.Weekly`
- `Todo.Recurrence.Monthly`
- `Todo.Recurrence.Weekdays`
- `Todo.Menu.Recurrence`

### 2. 状态文案建议

建议增加一个 `RepeatSummaryText` 或类似只读属性，用于拼接展示：

- `每天`
- `每周`
- `每月`
- `工作日`

然后与原本的截止时间状态组合：

- `每天 · 今天 18:00`
- `每周 · 2026/7/16 09:30`

### 3. 不建议的展示

第一阶段不建议：

- 在每条任务上再加一排标签
- 单独做重复任务筛选
- 做复杂的系列管理面板

这些都会让目前清爽的 Todo 变厚。

## 九、代码落点建议

### 1. 数据层

- `src/DeskBox/Models/TodoItem.cs`
- 新增 `TodoRecurrence.cs`
- `src/DeskBox/Models/TodoWidgetData.cs`
- `src/DeskBox/Services/TodoWidgetStore.cs`

### 2. 逻辑层

建议新增一个纯计算服务：

`src/DeskBox/Services/TodoRecurrenceService.cs`

职责：

- 判断是否重复任务
- 计算下一次截止时间
- 复制下一条任务模型

不要把这部分全塞进 `TodoWidgetViewModel`，否则完成逻辑会越堆越厚。

### 3. ViewModel

`TodoWidgetViewModel` 建议新增：

- `SetRecurrenceAsync(string itemId, string mode)`
- 在 `SetCompletedAsync()` 中接入“完成后生成下一条”

### 4. UI

`TodoWidgetContent.xaml.cs`

建议在右键菜单里新增：

- `重复` 子菜单

并在日期状态更新时接入重复文案展示。

## 十、边界与风险

### 1. 完成一条重复任务会生成更多已完成记录

这会让完成列表增长更快。

这是可接受代价，因为：

- 你当前已经有“清除已完成”
- 保留完成记录比直接覆盖更符合用户预期

### 2. 逾期补算逻辑必须统一

如果有些地方按“只加一步”，有些地方按“推进到未来”，会出现难以理解的差异。

所以要把计算统一封进 `TodoRecurrenceService`。

### 3. 月重复边界容易出错

`31 号`、`2 月`、闰年这些都要补测试。

### 4. Direct / Store 渠道无额外差异

重复任务是纯本地数据与本地逻辑，不涉及渠道差异，风险较低。

## 十一、测试建议

第一阶段至少覆盖以下测试：

### 1. 规则计算

- 每天推进
- 每周推进
- 每月推进
- 工作日推进
- 月末推进
- 逾期任务完成后推进到未来

### 2. 数据迁移

- 老版本 `todo.json` 无 `Recurrence` 时正常加载
- 清除截止日期时自动清除重复规则
- 非法 `Mode` 自动归一

### 3. 交互

- 右键设置重复模式
- 完成后立即生成下一条
- 新旧任务状态正确
- 提醒字段正确重置
- 文案展示正确

## 十二、推荐实施顺序

1. 先加数据模型与 Normalize
2. 再写 `TodoRecurrenceService`
3. 接入 `SetCompletedAsync`
4. 接右键菜单与本地化
5. 最后补状态文案展示与测试

## 十三、最终建议

重复任务这块，第一阶段最重要的不是“规则多”，而是三件事：

1. `入口轻`
2. `完成后推进稳定`
3. `不生成脏的下一次任务`

只要这三点做好，DeskBox 的 Todo 就会从“桌面轻待办”迈到“可以长期养成使用习惯”的阶段，而且不会破坏你现在这套轻量原生的方向。
