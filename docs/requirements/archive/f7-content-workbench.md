# DeskBox F7 内容工作台需求说明

状态：规划稿
创建日期：2026-06-19
适用版本：非 AI 第一阶段

## 1. 背景

DeskBox 当前最有价值的高频动作是：按下 F7，从屏幕右侧唤出格子，把格子里的文件拖到网页、微信、应用窗口中上传，或把任意文件夹里的文件拖回格子收纳。

这个行为已经超出了传统“桌面整理”的范畴。它本质上是一个跨应用操作台：用户正在微信、浏览器、资源管理器、办公软件之间切换时，需要一个随叫随到、用完即走的临时空间。

Windows 原生能力在以下场景存在明显断点：

- 便签只能写内容，不能很好地参与跨应用拖拽流。
- 剪贴板历史只能保存复制过的内容，缺少来源、状态和工作包概念。
- Microsoft To Do 偏任务管理，不适合和文件、网页上传、微信回复混合操作。
- 文件资源管理器能管理文件，但不能把文本、链接、待办、常用话术和文件放在同一个操作面板中。
- 网页上传、微信发送、表单填写都依赖用户在多个窗口之间来回切换。

因此第一阶段不接入 AI，也不做传统笔记/代办软件，而是扩展 DeskBox 的现有格子体验，增加“文本、待办、链接、话术、工作包”等内容格，让 F7 面板成为用户处理当下事务的轻量工作台。

## 2. 产品定位

### 2.1 一句话定位

DeskBox F7 内容工作台是一个 Windows 原生没有的跨应用内容暂存与操作面板，用于快速捕获选中文本、文件、链接和临时任务，并把它们以拖拽或复制的方式投递到微信、网页、文件夹和办公软件中。

### 2.2 核心原则

- 快速：F7 唤出后，用户应在 1 秒内完成捕获、拖入或拖出。
- 轻量：不做完整知识库，不要求用户先建立复杂分类体系。
- 原生：文件仍然是文件，文本仍然可以变成纯文本，拖拽行为符合 Windows 习惯。
- 可控：自动识别只能作为推荐，用户始终可以一键改类型。
- 临时优先：优先解决“现在要处理什么”，而不是“永久保存所有知识”。
- 低打扰：面板用完即走，不长期占用用户注意力。

### 2.3 非目标

- 第一阶段不接入 AI。
- 不做类 Notion 的文档数据库。
- 不做完整项目管理系统。
- 不做复杂日历和重复任务。
- 不做跨设备同步。
- 不做富文本编辑器。
- 不替代微信、浏览器、资源管理器或 Microsoft To Do。

## 3. 目标用户与痛点

### 3.1 目标用户

- 经常在微信、浏览器、文件夹之间传文件的人。
- 经常从聊天记录、网页、文档里摘取文字的人。
- 经常要把一段固定话术、地址、说明、报价发给别人的人。
- 经常需要处理“几份文件 + 一段说明 + 一个待办”的小任务包的人。
- 不想打开完整笔记软件或代办软件，只想快速记下并处理当下事务的人。

### 3.2 高频痛点

- 微信里看到一句事，想记成待办，但打开待办软件太重。
- 网页上选中一段资料，想临时保存并带上来源。
- 上传材料时，文件、说明文字、检查清单分散在不同地方。
- 常用话术散落在微信收藏、备忘录、文档里，发出时仍要复制粘贴。
- 复制一段内容后，剪贴板会被下一次复制覆盖。
- 想把多个文本片段整理后发给别人，但不想进入完整编辑器。

## 4. 核心概念

### 4.1 内容格

内容格是除文件格以外的轻量信息单元。第一阶段建议支持以下类型：

| 类型 | 用途 | 典型内容 | 拖出行为 |
| --- | --- | --- | --- |
| 文本格 | 临时记录普通文字 | 摘录、备注、报错、地址 | 输出纯文本 |
| 待办格 | 记录需要处理的动作 | 明天下午发合同 | 输出任务标题，可附状态 |
| 话术格 | 保存常用可复用文本 | 客服回复、地址、介绍 | 输出纯文本 |
| 链接格 | 保存 URL 和标题 | 网页、飞书文档、网盘链接 | 输出 URL 或标题 + URL |
| 清单格 | 保存多项检查项 | 上传前检查、资料清单 | 输出多行文本 |
| 文件格 | 现有文件收纳能力 | 文件、图片、PDF、压缩包 | 原生文件拖拽 |

### 4.2 收件箱

收件箱是所有快速捕获内容的默认入口。用户不需要先决定放在哪里，按 F7 捕获后先进入收件箱，再在需要时拖入其他格子或工作包。

### 4.3 今日

今日是当前要处理内容的临时队列。它不是完整日程，而是一个轻量状态区：

- 今天要发的文件。
- 今天要回复的消息。
- 今天要上传的材料。
- 今天要完成的几条待办。

完成后内容可以沉入“已完成”或回到原格子。

### 4.4 工作包

工作包是一组为了完成同一件事而临时组合的内容，可以包含：

- 文件。
- 文本说明。
- 待办清单。
- 链接。
- 常用话术。

示例：

```text
工作包：客户 A 报价资料
- 报价单.xlsx
- 产品图 3 张
- 待办：确认最终价格
- 话术：发送报价说明
- 链接：客户后台上传页面
```

工作包不是项目管理系统，而是“一次任务的材料盒”。

## 5. 主流程设计

### 5.1 F7 唤出

触发：用户按下 F7。

面板行为：

- 从屏幕右侧滑出，保持当前 DeskBox 唤起习惯。
- 面板不抢占用户当前窗口内容，视觉上覆盖但不打断。
- 若用户当前有选中文本，顶部出现捕获预览条。
- 若用户正在拖拽文件或文本，面板进入接收状态。
- 再次按 F7 或点击面板外区域，面板收起。

面板区域建议：

```text
顶部：捕获预览 / 搜索 / 快速新建
中部：收件箱、今日、固定格组
底部：最近使用 / 已完成入口 / 设置
```

### 5.2 选中文本后按 F7 捕获

用户场景：

用户在微信、网页、Word、PDF 或其他应用中选中一段文字，按 F7。

系统行为：

1. 尝试读取当前选中文本。
2. 成功读取后，在 F7 面板顶部显示捕获卡片。
3. 使用本地规则判断推荐类型。
4. 用户可以直接按 Enter 保存，也可以拖入某个格子。
5. 保存后默认进入收件箱，并保留来源信息。

捕获卡片内容：

- 文本前 2-3 行预览。
- 推荐类型，例如“可能是待办”“可能是链接”“保存为笔记”。
- 来源应用，例如微信、浏览器、Word。
- 捕获时间。
- 操作按钮：保存、改为待办、改为话术、取消。

技术注意：

- 若通过临时复制方式获取选中文本，必须尽量恢复用户原剪贴板内容。
- 如果获取失败，显示“无法读取选中内容，可复制后再按 F7 保存剪贴板内容”。
- 不应在用户不知情的情况下长期监听键盘输入。

### 5.3 拖入文本到格子

用户场景：

用户从网页、微信、文档中选中文字，直接拖到 DeskBox 内容格。

系统行为：

- 支持标准文本拖入。
- 拖入收件箱区域时创建文本格。
- 拖入今日区域时创建今日文本格。
- 拖入待办格组时可创建待办格。
- 拖入工作包时成为工作包内的文本项。

反馈：

- 目标区域高亮。
- 松手后出现轻量保存反馈。
- 不弹出阻塞式对话框。

### 5.4 拖出内容到微信或网页输入框

用户场景：

用户按 F7，拖一个笔记格、待办格或话术格到微信输入框。

系统行为：

- 文本类内容拖出时以纯文本投递。
- 链接格默认投递 URL。
- 清单格投递多行文本。
- 多选内容拖出时按顺序合并为一段文本。
- 文件格保持原生文件拖拽。

默认格式：

```text
文本格：原文
待办格：任务标题
清单格：
- 项目 1
- 项目 2
- 项目 3
链接格：标题 URL
```

可选设置：

- 待办拖出是否带状态。
- 链接拖出仅 URL 或标题 + URL。
- 多选拖出时是否用空行分隔。

### 5.5 从文件夹拖文件到工作包

用户场景：

用户把一批文件拖进一个工作包，准备上传或发送。

系统行为：

- 工作包内显示文件列表。
- 支持继续添加文本说明和检查清单。
- 文件拖出时保持现有 DeskBox 文件拖拽能力。
- 工作包可以被固定到今日。

关键体验：

- 用户不需要新建真实文件夹也能临时组合材料。
- 若文件来自现有路径，第一阶段优先用引用方式，不主动移动源文件。
- 对于收纳格中的文件，仍遵守现有移动/复制设置。

### 5.6 快速新建内容

入口：

- F7 面板顶部“+”按钮。
- 空白区域右键菜单。
- 快捷键，例如 `Ctrl + N`。

类型：

- 新建文本。
- 新建待办。
- 新建话术。
- 新建清单。
- 新建工作包。

交互要求：

- 默认焦点在标题或正文输入框。
- 输入完成后按 Enter 保存。
- `Esc` 取消。
- 支持粘贴图片或文件时自动转成对应格子。

### 5.7 今日处理流

用户场景：

用户把今天要做的内容集中到今日区域。

系统行为：

- 内容格右键或拖拽可加入今日。
- 今日区域显示未完成内容。
- 待办格可直接打勾完成。
- 文件和文本格可标记“已处理”。
- 已完成内容自动折叠到今日底部。

不做：

- 不做复杂日期视图。
- 不做周/月计划。
- 不做重复任务。

### 5.8 来源追踪

每个捕获内容尽量记录来源：

- 来源应用名称。
- 来源窗口标题。
- 来源 URL。
- 文件路径。
- 捕获时间。

显示原则：

- 卡片上只显示轻量来源，例如“来自微信 · 14:32”。
- 详细来源放在详情面板。
- 拖出时默认不带来源，避免污染发送内容。
- 用户可以从右键菜单选择“复制正文和来源”。

### 5.9 搜索与最近使用

F7 面板顶部支持搜索。

搜索范围：

- 标题。
- 正文。
- URL。
- 文件名。
- 标签。

最近使用：

- 最近拖出的内容。
- 最近捕获的内容。
- 最近打开的工作包。

交互：

- 按 F7 后直接输入关键词时，搜索框获得焦点。
- 上下键选择结果。
- Enter 打开或复制。
- 可将常用话术置顶。

## 6. 自动类型识别规则

第一阶段不接入 AI，只使用本地规则和启发式判断。

### 6.1 判断为链接格

满足任一条件：

- 内容是单个 URL。
- 内容包含 URL 且文字较短。
- 从浏览器地址栏或网页链接拖入。

保存内容：

- URL。
- 页面标题，若可获得。
- 来源浏览器窗口标题。

### 6.2 判断为待办格

命中多个信号时推荐为待办：

- 包含明确动作词：发、改、确认、提醒、提交、上传、下载、处理、联系、回复、整理。
- 包含时间词：今天、明天、下午、周一、月底、下周、几点。
- 文本较短，通常少于 120 字。
- 来自聊天窗口。

示例：

```text
明天下午把合同发我一下，顺便确认报价。
```

推荐结果：

```text
待办：明天下午发合同并确认报价
```

注意：

- 无 AI 阶段不拆分复杂多任务，只推荐整体为待办。
- 若要拆分成多条，用户可以手动点“拆成清单”。

### 6.3 判断为清单格

满足任一条件：

- 多行文本。
- 行首包含 `-`、`*`、数字序号、勾选符号。
- 内容包含多个顿号、分号，且结构明显。

系统行为：

- 保留原始文本。
- 可一键转为可勾选清单。

### 6.4 判断为话术格

满足任一条件时推荐保存为话术：

- 用户拖入固定话术格组。
- 文本以“您好”“麻烦”“感谢”“请查收”等沟通语气开头。
- 文本长度适中，适合重复发送。
- 用户连续多次拖出相同文本后，提示“是否固定为话术”。

### 6.5 默认判断为文本格

无法可靠判断时，默认保存为文本格。

原则：

- 宁可少猜，不要误分类。
- 自动识别只改变默认推荐，不隐藏其他选项。
- 用户修改类型后，系统记住同类来源的偏好。

## 7. 内容格详情设计

### 7.1 卡片视图

卡片应显示：

- 类型图标。
- 标题或首行文本。
- 一到两行预览。
- 来源与时间。
- 状态，例如今日、已完成、固定。

卡片不应显示过多说明文字。F7 面板是操作面板，不是阅读页面。

### 7.2 详情面板

双击或按 Enter 打开详情。

详情包括：

- 标题。
- 正文。
- 类型。
- 来源。
- 标签。
- 所属工作包。
- 创建时间和更新时间。
- 操作历史中的最后一次拖出时间。

### 7.3 右键菜单

建议菜单项：

- 复制正文。
- 复制正文和来源。
- 拖出格式。
- 加入今日。
- 标记完成。
- 固定。
- 移动到工作包。
- 改为文本 / 待办 / 话术 / 链接 / 清单。
- 在来源中打开。
- 删除。

## 8. 工作包设计

### 8.1 创建方式

- 手动新建工作包。
- 多选内容后右键“创建工作包”。
- 拖一个内容到另一个内容上，提示“创建工作包”。
- 从今日区域创建“今日工作包”。

### 8.2 工作包视图

工作包中按类型分组：

```text
标题：客户 A 资料上传

文件
- 报价单.xlsx
- 产品图 01.png

说明
- 发送话术

待办
- 确认报价
- 上传后截图给客户

链接
- 上传后台
```

### 8.3 工作包拖出

拖出到文件区域：

- 仅投递文件。

拖出到文本输入框：

- 投递工作包摘要。

拖出到微信：

- 如果含文件，优先拖出文件。
- 如果用户从文本摘要区域拖出，则输出纯文本。

复制摘要格式：

```text
客户 A 资料上传

文件：
- 报价单.xlsx
- 产品图 01.png

待办：
- 确认报价
- 上传后截图给客户
```

## 9. 面板布局建议

### 9.1 默认布局

```text
┌──────────────────────────────┐
│ 搜索 / 快速输入               │
├──────────────────────────────┤
│ 捕获预览：选中文本             │
├──────────────────────────────┤
│ 今日                          │
│ [待办] [文件] [话术]           │
├──────────────────────────────┤
│ 收件箱                        │
│ [文本] [链接] [清单]           │
├──────────────────────────────┤
│ 固定格组                      │
│ 常用话术 / 上传材料 / 临时文件 │
└──────────────────────────────┘
```

### 9.2 视觉原则

- 仍然保持 DeskBox 轻量、清爽、桌面浮层感。
- 卡片密度要比传统便签高，适合快速扫描。
- 卡片圆角与现有格子保持一致，不做过重装饰。
- 类型依赖小图标和轻量标签，不用大面积彩色块。
- 今日区域可以稍微突出，但不应变成任务看板。

### 9.3 空状态

收件箱空时：

```text
选中文本后按 F7，可快速保存到这里。
```

今日空时：

```text
把今天要处理的内容拖到这里。
```

文案应短，避免像功能介绍页。

## 10. 快捷键

建议支持：

| 快捷键 | 行为 |
| --- | --- |
| F7 | 唤出 / 收起面板 |
| Enter | 保存捕获预览或打开选中项 |
| Esc | 取消捕获或收起详情 |
| Ctrl + N | 新建文本格 |
| Ctrl + Shift + T | 新建待办格 |
| Ctrl + F | 搜索 |
| Delete | 删除选中内容 |
| Space | 待办完成 / 取消完成 |
| Ctrl + C | 复制正文 |
| Ctrl + Shift + C | 复制正文和来源 |

注意：

- 所有快捷键只在 F7 面板获得焦点时生效。
- 全局快捷键仍以 F7 为核心，不扩散过多系统级快捷键。

## 11. 数据模型建议

### 11.1 ContentItem

```json
{
  "id": "content_001",
  "type": "text",
  "title": "报价说明",
  "body": "您好，报价单已整理好...",
  "source": {
    "appName": "WeChat",
    "windowTitle": "文件传输助手",
    "url": null,
    "filePath": null,
    "capturedAt": "2026-06-19T15:30:00+08:00"
  },
  "status": "active",
  "isPinned": false,
  "isInToday": true,
  "tags": ["客户", "报价"],
  "workspaceId": null,
  "createdAt": "2026-06-19T15:30:00+08:00",
  "updatedAt": "2026-06-19T15:30:00+08:00"
}
```

### 11.2 ContentType

```text
text
todo
snippet
link
checklist
fileRef
workspace
```

### 11.3 存储建议

第一阶段建议使用本地 JSON 或 SQLite。

要求：

- 数据保存在 `%LocalAppData%\DeskBox\data`。
- 文件引用不复制文件本体，除非用户拖入收纳格。
- 删除内容格不删除源文件。
- 删除工作包不删除工作包内引用的源文件。
- 文本内容需要可导出，避免用户产生数据被锁住的感觉。

## 12. 与现有 DeskBox 的关系

### 12.1 保留现有文件格

现有文件格仍是核心，不应被内容工作台替代。

文件格负责：

- 文件收纳。
- 文件夹映射。
- 文件拖入拖出。
- 上传场景。

内容工作台负责：

- 文本捕获。
- 待办捕获。
- 常用话术。
- 链接保存。
- 文件和文本的临时组合。

### 12.2 组件形态

建议第一阶段不要做开放式插件，而是做内置内容组件：

- 收件箱组件。
- 今日组件。
- 话术组件。
- 工作包组件。

这些组件可以使用现有 WidgetWindow 外壳，但内容区域使用专门的 WinUI 控件。

### 12.3 F7 行为兼容

现有 F7 行为不应被破坏：

- 用户只想拖文件上传时，仍然能看到原有文件格。
- 内容工作台可以作为右侧面板中的一个区域，而不是完全替代所有格子。
- 若用户不启用内容工作台，DeskBox 仍按现在方式运行。

### 13.1 P0 必做

- F7 面板支持收件箱和今日区域。
- 支持新建文本格、待办格、话术格、链接格、清单格。

- 支持文本类内容拖出为纯文本。
- 支持拖入文本创建内容格。
- 支持内容格加入今日、标记完成、删除。

- 支持搜索标题和正文。
- 支持本地持久化。

- 支持多选内容拖出合并文本。

## 14. 关键交互验收标准

### 14.1 选中文本捕获

验收：

- 在微信中选中一段文字，按 F7 后能看到捕获预览。
- 按 Enter 后内容保存到收件箱。
- 内容能再次拖到微信输入框，变成纯文本。
- 捕获失败时有明确提示，不破坏用户剪贴板。

### 14.2 话术复用

验收：

- 用户创建一条话术格。
- 按 F7 后能在固定话术区域找到。
- 拖到微信输入框后输出纯文本。
- 话术不会自动附带来源或内部标签。

### 14.3 今日处理

验收：

- 任意内容格可加入今日。
- 待办格可打勾完成。
- 已完成内容折叠显示。
- 今日清空不会删除原始内容，除非用户明确删除。

### 14.4 工作包

验收：

- 工作包内可同时包含文件和文本。
- 文件可继续拖到网页上传。
- 文本说明可拖到微信或网页输入框。
- 删除工作包不删除源文件。

## 15. 边界与风险

### 15.1 选中文本读取不稳定

不同应用对选中文本暴露能力不同。第一阶段需要准备多路径：

- 优先使用 UI Automation 获取选中文本。
- 失败时尝试临时复制。
- 再失败时提示用户复制后保存剪贴板。

必须避免：

- 破坏用户剪贴板。
- 在密码框或敏感输入区域误读取。
- 因读取失败让 F7 面板卡顿。

### 15.2 拖出到不同应用差异

微信、浏览器输入框、Word、文件资源管理器对拖拽数据格式支持不同。

策略：

- 文本拖出提供 Unicode Text 和 Plain Text。
- 文件拖出沿用原生文件拖拽格式。
- 如果目标应用不接受拖拽文本，提供一键复制作为兜底。

### 15.3 数据隐私

内容格可能保存聊天文字、客户资料、地址、合同信息。

第一阶段要求：

- 默认只本地保存。
- 不上传云端。
- 不接入 AI。
- 明确提供一键清空收件箱。
- 支持彻底删除。

## 16. 推荐迭代路线

### 第一迭代：文本捕获闭环

目标：让用户第一次感受到“选中文字 -> F7 -> 保存 -> 拖出去发给别人”的价值。

范围：

- 收件箱。
- 文本格。
- 待办格。
- 话术格。
- 选中文本捕获。
- 纯文本拖出。

### 第二迭代：今日与状态

目标：让用户把 DeskBox 当作当天事务处理台。

范围：

- 今日区域。
- 完成状态。
- 搜索。
- 最近使用。
- 来源记录。

### 第三迭代：工作包

目标：把文件上传和文字说明合并到同一操作流。

范围：

- 工作包。
- 文件引用。
- 多选拖出。
- 工作包摘要复制。

### 第四迭代：高级捕获

目标：进一步覆盖 Windows 原生没有的捕获体验。

范围：

- 剪贴板历史转内容格。
- 截图 OCR。
- 链接标题获取。
- 规则识别优化。

## 17. 产品判断

这个功能最重要的价值不是“多一个笔记功能”或“多一个待办功能”，而是补上 Windows 桌面工作流中缺失的一层：

```text
我正在任何应用里看到一个东西
按 F7
它被收进一个可操作的格子
之后我可以拖出去、发出去、上传出去、处理掉
```

只要这个闭环足够顺，DeskBox 就会从“文件收纳工具”升级成“跨应用内容操作台”。这件事 Windows 原生没有做好，传统笔记和待办软件也没有贴着这个场景做。
